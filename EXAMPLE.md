# Account Exposure Worked Examples

Snapshot date: 2026-06-13

This document uses real values from the current large synthetic dataset generated by:

```bash
PYTHONPATH=src python -m screensmart.exposure.synthetic_graph --seed 42 --reset
PYTHONPATH=src python -m screensmart.exposure.precompute --max-depth 3 --top-k 20 --reset
```

The goal is to show the exact computation path the module currently uses, so the
graph behavior can be audited against real rows instead of hand-wavy examples.

## Scoring Formula

The current module computes exposure offline with these rules:

```text
source_risk:
  SANCTIONED = 1.0
  SUSPICIOUS = 0.7

flow_materiality_weight =
  0.70 * amount_weight(total_amount)
  + 0.30 * concentration_score(max(outgoing_concentration, incoming_concentration))

edge_score =
  relation_weight(edge_type)
  * flow_materiality_weight
  * time_decay(last_seen)
  * confidence

new_score =
  current_score
  * edge_score
  * hop_decay(next_depth)
```

Current lookup threshold:

```text
REVIEW_THRESHOLD = 0.08
```

Important behavior:

- direct sanctioned IBAN/account => `MATCH`
- score `>= 0.08` => `REVIEW`
- sanctioned path within 2 hops can still force `REVIEW` even if score is below `0.08`
- weak historical tiny flows can be dropped during precompute and never reach `exposure_index`
- for `USES_ACCOUNT` / `OWNS` / `SHARED_IDENTIFIER`, materiality stays `1.0`

```text
GB72SS42000000000000000151 (SANCTIONED IBAN)
  --SENT_TO amount=23000 last_seen=2026-06-10 confidence=0.94-->
RS73SS42000000000000000152 (recipient)
```

Exact computation:

```text
source_risk = 1.0

relation_weight(SENT_TO) = 0.6
amount_weight(23000) = 0.8
outgoing_concentration = 1.0
incoming_concentration = 1.0
concentration_score(1.0) = 1.0
flow_materiality_weight = 0.70 * 0.8 + 0.30 * 1.0 = 0.86
time_decay(2026-06-10) = 1.0
confidence = 0.94

edge_score = 0.6 * 0.86 * 1.0 * 0.94 = 0.48504
hop_decay(depth=1) = 0.7

new_score = 1.0 * 0.48504 * 0.7 = 0.339528
stored_score = 0.3395
```

Returned signal:

```json
{
  "module": "account_exposure",
  "recommended_action": "REVIEW",
  "score": 0.3395,
  "source_risk_level": "SANCTIONED",
  "source_risk_node": "GB72SS42000000000000000151",
  "best_depth": 1,
  "rule_triggers": ["SCORE_AT_OR_ABOVE_THRESHOLD"],
  "reason": "REVIEW: recipient account has exposure score 0.3395 from sanctioned anchor GB72SS42000000000000000151 at depth 1 via SENT_TO."
}
```

Validation note:
This also makes sense. It is recent, high enough in amount, and only one hop away.

## 3. Two-Hop Sanctioned Override

Source payment row:

```text
case_id: two_hop_exposure-s42-0352
recipient_name: Omar Khalaf Logistics Payee
recipient_iban: TR32SS42000000000000000556
expected_verdict: REVIEW
amount: 4895 EUR
```

Best path stored in `exposure_index`:

```text
IT30SS42000000000000000554 (SANCTIONED IBAN)
  --SENT_TO amount=17800 last_seen=2026-06-04 confidence=0.88-->
TR31SS42000000000000000555
  --SENT_TO amount=17075 last_seen=2026-06-07 confidence=0.84-->
TR32SS42000000000000000556 (recipient)
```

Exact computation:

```text
source_risk = 1.0

Step 1:
relation_weight(SENT_TO) = 0.6
amount_weight(17800) = 0.8
outgoing_concentration = 1.0
incoming_concentration = 1.0
concentration_score(1.0) = 1.0
flow_materiality_weight_1 = 0.70 * 0.8 + 0.30 * 1.0 = 0.86
time_decay(2026-06-04) = 0.8
confidence = 0.88
edge_score_1 = 0.6 * 0.86 * 0.8 * 0.88 = 0.363264
hop_decay(depth=1) = 0.7
score_after_step_1 = 1.0 * 0.363264 * 0.7 = 0.2542848

Step 2:
relation_weight(SENT_TO) = 0.6
amount_weight(17075) = 0.8
outgoing_concentration = 1.0
incoming_concentration = 1.0
concentration_score(1.0) = 1.0
flow_materiality_weight_2 = 0.70 * 0.8 + 0.30 * 1.0 = 0.86
time_decay(2026-06-07) = 1.0
confidence = 0.84
edge_score_2 = 0.6 * 0.86 * 1.0 * 0.84 = 0.43344
hop_decay(depth=2) = 0.4
score_after_step_2 = 0.2542848 * 0.43344 * 0.4 = 0.0440816265216
stored_score = 0.0441
```

Decision:

```text
0.0441 < 0.08
but best_depth = 2
and path reaches a SANCTIONED source
so final action = REVIEW
```

Returned signal:

```json
{
  "module": "account_exposure",
  "recommended_action": "REVIEW",
  "score": 0.0441,
  "source_risk_level": "SANCTIONED",
  "best_depth": 2,
  "rule_triggers": ["SANCTIONED_PATH_WITHIN_2_HOPS"],
  "reason": "REVIEW: recipient account is 2 hops from a sanctioned account via SENT_TO -> SENT_TO; routed to review despite low decayed exposure score 0.0441."
}
```

Validation note:
This is one of the strongest sanity checks in the current system. The score is low
because of hop decay, but the override still preserves the intended alert.

Source payment row:

```text
case_id: shell_company-s42-1001

COMPANY:s42:shell-company:0001 (SUSPICIOUS COMPANY)
  --USES_ACCOUNT confidence=0.99-->
CY70SS42000000000000001751 (recipient)
```

Exact computation:

```text
source_risk = 0.7

relation_weight(USES_ACCOUNT) = 0.9
amount_weight is forced to 1.0 for USES_ACCOUNT
time_decay(2026-06-08) = 1.0
confidence = 0.99

edge_score = 0.9 * 1.0 * 1.0 * 0.99 = 0.891
hop_decay(depth=1) = 0.7

new_score = 0.7 * 0.891 * 0.7 = 0.43659
stored_score = 0.4366
```

Returned signal:

```json
{
  "module": "account_exposure",
  "recommended_action": "REVIEW",
  "score": 0.4366,
  "source_risk_level": "SUSPICIOUS",
  "source_node_type": "COMPANY",
  "rule_triggers": ["SCORE_AT_OR_ABOVE_THRESHOLD"]
}
```

Validation note:
This behavior is coherent with the current algorithm, but it exposes an important
modeling choice: the best source is the suspicious shell company itself, not the
upstream sanctioned owner. That happens because precompute keeps the best score per
node, and the direct `USES_ACCOUNT` edge from a suspicious company scores higher
than a weaker two-hop path from the sanctioned owner.

## 5. High-Volume Proxy

Source payment row:

```text
case_id: high_volume_proxy-s42-1151
recipient_name: Ilham Haddad Proxy Route
recipient_iban: RS44SS42000000000000001903
expected_verdict: REVIEW
amount: 15000 EUR
```

Best path stored in `exposure_index`:

```text
RS43SS42000000000000001902 (SUSPICIOUS IBAN)
  --SENT_TO amount=47000 last_seen=2026-06-12 confidence=0.96-->
RS44SS42000000000000001903 (recipient)
```

Surrounding sender and recipient behavior in the current generated graph:

```text
Outgoing from RS43SS42000000000000001902:
  -> RS44SS42000000000000001903 amount=47000
  -> GB48SS42000000000000001907 amount=6500
  -> CH47SS42000000000000001906 amount=3000
  -> ES45SS42000000000000001904 amount=2000
  -> IT46SS42000000000000001905 amount=1500
  sender_total_outgoing_amount = 60000

Incoming to RS44SS42000000000000001903:
  <- RS43SS42000000000000001902 amount=47000
  <- IT51SS42000000000000001910 amount=4000
  <- CH50SS42000000000000001909 amount=3500
  <- GB49SS42000000000000001908 amount=2500
  receiver_total_incoming_amount = 57000
```

Exact computation:

```text
source_risk = 0.7

relation_weight(SENT_TO) = 0.6
amount_weight(47000) = 0.8
sender_total_outgoing_amount = 60000
receiver_total_incoming_amount = 57000
outgoing_concentration = 47000 / 60000 = 0.783333
incoming_concentration = 47000 / 57000 = 0.824561
flow_concentration = max(0.783333, 0.824561) = 0.824561
concentration_score(0.824561) = 1.0
flow_materiality_weight = 0.70 * 0.8 + 0.30 * 1.0 = 0.86
time_decay(2026-06-12) = 1.0
confidence = 0.96

edge_score = 0.6 * 0.86 * 1.0 * 0.96 = 0.49536
hop_decay(depth=1) = 0.7

new_score = 0.7 * 0.49536 * 0.7 = 0.2427264
stored_score = 0.2427
```

Returned signal:

```json
{
  "module": "account_exposure",
  "recommended_action": "REVIEW",
  "score": 0.2427,
  "source_risk_level": "SUSPICIOUS",
  "source_risk_node": "RS43SS42000000000000001902",
  "rule_triggers": ["SCORE_AT_OR_ABOVE_THRESHOLD", "HIGH_COUNTERPARTY_CONCENTRATION"]
}
```

Validation note:
This is another place where the current algorithm favors the strongest local source.
The synthetic story is "sanctioned source -> suspicious proxy -> recipient", but the
stored best explanation is the suspicious proxy one hop away, because that path wins
on score.

Why materiality is better than absolute amount alone:

- absolute amount alone says `47000` is large
- concentration shows this edge is also about `78%` of the sender's outgoing flow and
  about `82%` of the receiver's incoming flow
- that makes the transfer behaviorally important, not just numerically large

Contrast note:

- the same `47000` amount coming from a hub with `5,000,000` total outgoing flow
  would have much lower concentration
- here it is materially stronger because it represents most of the proxy account's
  outgoing behavior

## 6. Old Tiny Exposure

Source payment row:

```text
case_id: old_tiny_exposure-s42-0551
recipient_name: Rashid Mansouri Legacy Counterparty
recipient_iban: DE94SS42000000000000001152
expected_verdict: NO_MATCH
amount: 2100 EUR
```

Relevant graph edge that exists in the raw graph:

```text
IT93SS42000000000000001151 (SANCTIONED IBAN)
  --SENT_TO amount=5 last_seen=2025-01-29 confidence=0.42-->
DE94SS42000000000000001152 (recipient)
```

Why precompute drops it:

```text
edge_type = SENT_TO
amount = 5
ignore_weak_edge = True
because edge_type == SENT_TO and amount < 100
```

Effect:

- recipient never gets an `exposure_index` row
- realtime lookup therefore returns `NO_EXPOSURE_INDEX_ENTRY`
- the edge is not scored under the final materiality formula because weak-edge pruning
  runs first
- concentration is intentionally not allowed to rescue dust-like historical transfers

Returned signal:

```json
{
  "module": "account_exposure",
  "recommended_action": "NO_MATCH",
  "score": 0.0,
  "rule_triggers": ["NO_EXPOSURE_INDEX_ENTRY"],
  "reason": "NO_MATCH: no exposure index entry found for recipient account."
}
```

Validation note:
This is logically fine, but the wording matters. The graph does contain a weak
historical link; it just gets filtered out before indexing.

## What These Examples Tell Us

The current generator and scoring are internally consistent. The one-hop, two-hop,
old-tiny, and clean cases all behave the way the formulas suggest they should.

The most important realism caveat is this:

- `shell_company` and `high_volume_proxy` often surface the nearest strongest
  suspicious anchor, not the ultimate sanctioned source

That is not a data-generation bug. It is a direct consequence of the current
precompute strategy:

- best score wins per node
- the stored explanation path is the best-scoring path, not the most semantically
  interesting path

If the product story needs "show me the sanctioned provenance even when a suspicious
intermediary gives a stronger local score", that is a future algorithm/output change,
not a documentation fix.
